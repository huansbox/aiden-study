## Problem Statement

二兒子（下稱弟弟）5.5 歲、幼稚園中班、下月升大班，**不識字**。家長想要一個像哥哥題庫網站那樣、在 iPad 上親子共學的練習工具，幫弟弟熟悉注音符號。家長明確的教學直覺：

- **聽音辨認**：發音「ㄅ」，在幾張字卡中選出「ㄅ」
- **練習發音**：出示字卡「ㄅ」，弟弟唸出來，家長評「熟練／不太熟／不會」（anki 精神）
- **小目標不是全部學會**，而是先學會一部分後**快速拿來拼字**——ㄅㄚ、ㄆㄚ、ㄇㄚ、ㄈㄚ、ㄅㄨ⋯搭配四聲調就有很多組合

現有哥哥版網站以識字兒童為前提（文字題幹、文字按鈕），弟弟完全用不了。市面 app 無法客製節奏、沒有親子共學評分、進度不在自己手上。

## Solution

在同 repo 新增一個**獨立的單檔網頁 app**（部署於現有網站的 `/zhuyin/` 子路徑），整個答題介面**聲音導向**：題目語音自動播、可重聽、大字卡點選、音效＋動畫回饋，弟弟不識字也能自己作答；外圍導航（選模式、看進度、收工）保留文字給家長。

北極星＝**最快讓弟弟體驗「我會拼字了」**（synthetic phonics 路線：教一小組符號就馬上合成真實詞），不是背完 37 個符號。v1 只做「最小拼讀圈」：ㄅㄆㄇㄈ＋ㄚㄨ＋四聲調，8 個符號就能拼出爸、媽、馬、木、斧⋯一批高情緒價值的真詞。

四種學習活動：**聽音辨認**（機器判分）→ **發音字卡**（家長三檔評分）→ **聽音組字**（播「ㄇㄚˇ」→選符號→選聲調，機器判分）→ **拼讀出聲**（看「ㄇㄚˇ」唸出來，家長評分）。搭配**示範卡**（新音節首次出現先教再考）與**真實詞連結**（拼對跳詞圖＋詞語音「馬！」）交付北極星時刻。

排程用**優先權佇列**（非日曆 SRS）：每批 5 題，防挫折設計（「不會」配額上限、批首暖身、批尾成功收尾）。進度存 localStorage 並**自動同步**到家庭進度服務（獨立 Cloudflare Worker＋KV），對抗 iOS 約 7 天清除 storage；哥哥版日後可接同一套服務。

## User Stories

**弟弟（5.5 歲學習者，不識字）**

1. 身為不識字的學習者，我想要題目用聲音出、字卡夠大顆，讓我不用認字也能自己作答
2. 身為學習者，我想要題目聲音自動播放、且有一顆大大的重聽鈕，讓我沒聽清楚時可以再聽
3. 身為學習者，我想要答對時有好聽的音效和動畫，讓我知道自己做對了、想繼續玩
4. 身為學習者，我想要答錯時不被難聽的聲音嚇到——錯的卡片變灰、題目自動重播、我可以在剩下的卡片裡繼續選到對為止，讓每一題都以成功結束
5. 身為學習者，我想要每個新音節第一次出現時先聽「ㄇ⋯ㄚ⋯ㄇㄚ」的分段慢速示範（不算分），讓我先學會「兩個音怎麼黏在一起」再被考
6. 身為學習者，我想要一顆「慢慢聽」鈕可以聽題目的分段慢速版，讓我聽得出音節裡面有哪些音
7. 身為學習者，我想要拼對音節時看到對應真實詞的圖並聽到「馬！」，讓我發現自己真的會拼字了
8. 身為學習者，我想要最早學的組合就能拼出「爸爸」「媽媽」，讓我拼出來的第一個詞就是最有感覺的詞
9. 身為學習者，我想要聽音組字分兩小步（先選符號、再選聲調），讓我一次只需要做一個決定
10. 身為學習者，我想要聲調那一步的四顆聲調鈕每顆都可以點來聽，讓我用「比較」代替「回想」
11. 身為學習者，我想要一批只有 5 題、批末有獎杯可以點開看獎勵插畫，讓一次練習短短的、結尾是開獎品的儀式感
12. 身為學習者，我不想要一整批都是我不會的卡，讓我每批至少答對大部分、不會越練越挫折
13. 身為學習者，我想要卡片之間留大間距、剛換題的短暫瞬間不吃點擊，讓我亂點或連點時不會誤答

**家長（共學教練、內容與進度的管理者）**

14. 身為家長，我想要開 app 後由我選模式和內容範圍、之後答題迴圈弟弟自主，讓共學時我掌控節奏但不用一直代操作
15. 身為家長，我想要發音卡上弟弟唸完後，用螢幕底緣三顆大色塊（不顯示「不會」等字樣）單手一拍完成評分並自動翻下一題，讓評分不打斷節奏、也不讓弟弟看懂自己被打了差評
16. 身為家長，我想要點下評分的同時 app 自動播一次正確發音，讓弟弟唸錯時立刻聽到對的、我不用自己示範
17. 身為家長，我想要排程自動把「不會」和「不太熟」的卡排優先、並固定抽查「熟練」的舊卡，讓複習不靠我的記憶
18. 身為家長，我想要同一張卡連續多批都失敗時 app 建議「先收起來過幾天再出」，讓演算法不跟 5 歲小孩硬拚
19. 身為家長，我想要新符號按教學設計的順序進場（ㄅ→ㄇ→ㄚ→ㄈ→ㄨ→ㄆ）、且開下一圈由我手動決定（app 只提示建議），讓難度升級的時機在我手上
20. 身為家長，我想要出題的目標聲調漸進（先一、四聲，穩了加二聲、最後三聲），讓最難分的二三聲不會在一開始就打擊信心
21. 身為家長，我想要一個進度總覽看到每個符號／音節在各活動的三檔狀態，讓我知道弟弟卡在哪
22. 身為家長，我想要批間畫面提供「再一批／今天到這裡」，讓收工的決定權在我、且不會自動連批
23. 身為家長，我想要進度自動同步到雲端、iPad 被 iOS 清掉儲存後打開 app 自動還原，讓我完全不用記得備份
24. 身為家長，我想要弟弟主畫面圖示的網址自帶密鑰（`?k=`），讓「點圖示」就等於登入他自己的存檔、密鑰不怕被清 storage
25. 身為家長，我想要家長區顯示「上次同步時間」，讓同步壞掉時看得出來而不是無聲失敗
26. 身為家長，我想要一個匯出／匯入文字框逃生門（含顯示同步密鑰），讓雲端服務出事或密鑰遺失時有手動退路
27. 身為家長，我想要 app 在音檔缺漏或同步服務掛掉時靜默降級、照常作答，讓弟弟的練習永遠不被技術問題中斷

**維護者（開發與內容擴充）**

28. 身為維護者，我想要符號、進場順序、音節組合、聲調階段、混淆對表、真實詞連結、音檔 key 全部定義在手寫內容資料檔，讓開新圈＝改資料不改程式
29. 身為維護者，我想要內容資料有不變量測試（讀真實出貨檔、聚合列出所有錯誤），讓手寫內容的筆誤在進 iPad 前被擋下
30. 身為維護者，我想要音檔有命名規約＋「每 key 有檔、無孤兒檔」的審計，讓親錄或批次生成兩條路長出來的檔案不會亂
31. 身為維護者，我想要核心邏輯全在可獨立測試的純函式區塊（沿用 repo sentinel 慣例），讓改排程或判分規則時有回歸保護
32. 身為維護者，我想要同步服務的 handler 可以用記憶體 KV stub 單元測試、且 client/server 的還原判定有契約測試對齊，讓兩端邏輯不會默默分岔

## Implementation Decisions

**部署與形態**
- 新 app＝現有網站 `/zhuyin/` 子路徑下的獨立單檔 HTML（內嵌 CSS/JS、零依賴），與哥哥版同 origin 但完全獨立的頁面與 localStorage key（`aiden_zhuyin_v1`）。網站**維持 GitHub Pages 部署不搬家**（本 PRD 代拍板：整站遷 Cloudflare Pages 解耦為後續獨立專案，見「待家長拍板」）
- **不掛 PWA manifest**：主畫面圖示走純 webclip，避免 `start_url` 吃掉 `?k=` 密鑰，也避開哥哥版踩過的 standalone/Safari 雙容器問題
- 哥哥版純函式資產（獎勵選圖、批間畫面 pattern 等）用**複製再改**重用，不抽共用模組；複製處加來源註解（出處區塊＋commit），讓日後 bugfix 可刻意雙向移植

**內容模型（資料驅動）**
- 手寫內容 JSON 定義：符號（字形、音檔 key）、**圈**（circle，一組符號＋進場序列）、音節組合表（聲母×韻母×聲調、音檔 key、可選的真實詞連結：詞、詞音檔、詞圖）、聲調解鎖階段、混淆對表（v1 只有 ㄅ↔ㄆ）
- v1 第一圈＝ㄅㄆㄇㄈ＋ㄚㄨ＋四聲；圈內進場序列＝**ㄅ→ㄇ→ㄚ→ㄈ→ㄨ→ㄆ**（ㄆ 最後進場、定位為「跟 ㄅ 比一比」的對比課；教學 reviewer：ㄅ/ㄆ 為聽覺＋視覺雙重最小對立，不可同時當新卡）
- 聲調漸進：出題目標音先只用一、四聲，達標後加二聲、最後三聲（兒童聲調習得順序）；聲調鈕 UI 恆為四顆可點聽。輕聲不進 v1（詞連結播「爸爸」時自然唸輕聲即可，不教）
- 真實詞連結：拼讀答對跳「詞圖＋詞語音」卡；爸爸、媽媽刻意排最早解鎖的組合；無幼兒友善真詞的音節不硬配

**四種學習活動＋示範卡**
- 聽音辨認：播符號音→四選一大字卡，機器判分。干擾項分層規則：目標為新卡／「不會」時強制遠對比（不同發音部位、不同形），目標「熟練」後才配混淆對表的最小對立當挑戰題；干擾項只從**已進場**符號抽；池不足 4 時降為三選一（不跨圈補位、不劇透未學符號）
- 發音字卡：出示符號→弟弟唸→家長底緣三色鈕一拍評分（熟練／不太熟／不會，色塊＋中性 icon、不顯字）；點評分自動播正確音並翻下一題
- 聽音組字：播完整音節（預設接一次分段慢速）→第一步依序選聲符、韻符（每選對一個立即播該符號音確認）→第二步四顆聲調鈕（各可點聽）選調。判分狀態機的中途語意（聲韻對聲調錯算哪檔、重試規則）於實作時以「狀態×事件」表完整枚舉並逐格測試
- 拼讀出聲：出示音節→弟弟唸→家長評分（重用發音卡機制換音節內容）。v1 保留但為四活動中優先級最低者（時程緊先砍它）
- 示範卡：每個新音節首次出現先走不判分的示範（「ㄇ⋯ㄚ⋯ㄇㄚ」分段連讀）；答題畫面常設「慢慢聽」鈕播分段版。分段慢速素材由音檔策略章的資產管線提供

**答錯回饋（error-correction loop）**
- 答錯＝中性音效（禁 buzzer／大紅叉）→錯卡變灰→自動重播題目→在剩餘選項續選至對（選對仍播完整正確回饋）→該題記為錯、排批尾複驗。每題皆以成功收尾

**排程（優先權佇列，非日曆 SRS）**
- 每批 5 題。組批規則：「不會」優先但**配額上限 2 格**；次「不太熟」；再新卡；**固定 1 格隨機抽查「熟練」**。批內排位：第 1 格放穩贏暖身（熟練抽查移此）、難卡中段、批尾保證成功收尾
- 同卡連續 3 批失敗→提示家長「先收起來，過幾天再出」（suspend，家長一鍵執行／忽略）
- 三檔狀態來源：家長評分活動直接寫檔；機器判分活動以答對答錯映射升降檔（映射規則實作時定於純函式並測試）
- 新圈解鎖＝家長手動，app 於現圈大致熟練時提示建議
- 隨機性（洗牌、抽查、干擾項）一律以 rng 函式注入純函式（repo 既有慣例），不引入 seeded PRNG

**獎勵**
- 照搬哥哥版「批末獎杯揭曉→點開隨機獎勵插畫」機制與洗牌袋；弟弟專屬圖池另由設計圖片 session 生成（池 key 綁圈／主題），生圖前先借 `__generic__` 通用池；缺圖靜默不顯示（既有容錯哲學）
- 真實詞插圖（10–15 張）與獎勵圖池同批委託生圖 session

**聲音（策略待 iPad spike 定案，見「待家長拍板」）**
- 兩條路：家長親錄（教學 reviewer 強烈傾向：單獨符號音、短促聲符＋分段慢速示範 TTS 做不出來；且爸媽聲音本身是 reward）或 Apple TTS 批次生成（Mac `say`）。v1 第一圈約 50–70 段（含分段慢速版與真詞語音）
- 不論哪條路，先定**音檔資產管線規格**：內容 JSON 音檔 key ↔ 檔名命名規約、資產目錄、批次生成／錄音輔助腳本、「每 key 有檔、無孤兒檔」審計（掛進內容不變量測試）
- 播放層＝薄 adapter：iOS 需 user gesture 解鎖音訊——「開始練習」鈕作為解鎖點，之後迴圈內程式化播放；此模式**必須在寫題型邏輯前於 iPad 真機驗證**（現有 spike 頁只驗了點按發聲，未驗自動播）。家長區加隱藏「音效自檢」鈕（逐類播一輪），把真機驗證壓到一分鐘
- 缺檔 fallback 鏈（如：分段慢速版缺→播常速；詞語音缺→只顯圖）為純函式可測

**進度同步（家庭進度服務）**
- 獨立 Cloudflare Worker＋KV（不與網站同部署）。資料模型：key＝`{child}:{app}`（弟弟 v1 即 `<child>:zhuyin`，child 識別字串待家長定，見拍板節），value＝整包存檔＋schemaVersion＋server 遞增版本號 rev
- API：GET（回存檔＋rev）／PUT（帶 client 已知 rev；server 接受即 rev+1 回傳）。**同步決策邏輯全程不碰時鐘**（兩 reviewer 對時間戳的分歧以 rev 計數器調和）；client 的 updatedAt 僅供「上次同步時間」顯示
- 密鑰＝隨機 token 放主畫面圖示 URL query（`?k=`），server 驗 token；token 同時顯示於家長區匯出逃生門旁（遺失復原路徑）
- Client 還原決策為顯式矩陣純函式（`decideSync`）：兩邊皆空／本地空＋遠端有→還原／遠端 404 或壞 payload **絕不可**視為空而 push 覆蓋／rev 落後且本地無新變更→取遠端／schemaVersion 比 app 新→拒讀拒 push 保留本地。每批結束靜默 push；離線或服務掛→靜默略過，app 完全不依賴同步也能跑
- schemaVersion 策略：server 是啞 blob store；migration 全在 client load 時做（舊→新）；KV 最終一致性為已知限制（單家庭用量可接受），不加 conditional-write
- 匯出／匯入文字框逃生門只 scope `aiden_zhuyin_v1`（同哥哥版慣例）
- 哥哥版接入同一服務＝後續另案，v1 不動哥哥版任何程式

**外圍 UX**
- 家長導航區：模式選擇、圈／聲調階段管理、三檔狀態總覽、上次同步時間、逃生門、音效自檢。批間畫面＝「再一批／今天到這裡」（哥哥版 pattern）
- 觸控防呆：卡片大間距、換題後約 400ms 忽略點擊、第一次有效點擊即判分（不設二次確認）

## Testing Decisions

**好測試的定義**：只測外部行為（輸入→輸出／狀態轉移），不測實作細節；隨機性與環境（rng、KV、request）以參數注入；DOM、音訊播放、動畫不進單元測試（沿 repo 慣例，交給手動 E2E）。

**接縫（3 個，經測試視角 reviewer 定稿）**

1. `<zhuyin-core-pure>` sentinel 區塊＋對應 node 測試檔：組批佇列（配額／暖身收尾排位／批尾複驗）、三檔狀態轉移與機器判分映射（狀態×事件全枚舉，密度比照國語 mode 純函式測試）、圈→題池展開（進場序列、聲調階段）、干擾項規則（遠對比／混淆對表／已進場池／池不足降級、rng 注入驗不變量：正解恰一次、無重複）、解鎖建議、suspend 判定、音檔 key fallback 鏈
2. `<zhuyin-sync-pure>` sentinel 區塊＋對應 node 測試檔：payload 組建、`decideSync` 完整決策矩陣（含壞 payload、schemaVersion 邊界）、URL 密鑰解析
3. 同步 Worker handler＝真 ES module 直接 import 測試（不用 sentinel）：in-memory KV stub（全 async）跑 GET／PUT／rev 落後拒絕／錯 token／壞 payload 不得回傳成可覆蓋狀態。**外加一支契約測試**：同一份 case 矩陣同時餵 client 的 `decideSync` 與 server handler，斷言兩端判定一致（防默默分岔）

**內容不變量**：掛 node 測試（此子專案內容不經任何 Python 環節，不開 Python validator——兩位 reviewer 獨立確認）；讀**真實出貨的**內容 JSON 做審計式斷言（每符號有音檔 key、音節組合合法、圈引用符號存在、混淆對表雙向、音檔 key 對得到資產），**聚合列出全部錯誤**不 fail-fast（authoring UX 比照國語 curated validator）

**先例（prior art）**：sentinel 抽取 eval＋node --test＝國語 mode／備份／換行／獎勵選圖四個既有測試檔；rng 注入＝獎勵選圖測試（`()=>0`／`()=>0.999`／洗牌袋序列驗證）；區塊紀律照舊（區塊內不得引用外部全域、頭部警告註解照抄）

**手動驗證（非接縫）**：Playwright 對部署站 E2E smoke（沿哥哥版慣例）＋ iPad 真機 checklist 明列：音訊手勢解鎖後自動播存活、主畫面圖示開啟後 `?k=` 仍在且同步可用、清 storage 後自動還原、音效自檢一輪。spike 結論落成 docs-dev 文件（比照備忘錄捷徑配方文件作為手動驗證真相源）

**驗證清單紀律**：repo 無 CI，新增的 node 測試檔記入子專案驗證 checklist（比照現況 pytest＋逐一 node --test）
