# 期末題庫擴充 — 實作經驗筆記（給未來 session 的 LLM）

> 目的：讓未來的 session 不必重踩坑就能接手維護或擴充。
> 狀態源看 `HANDOFF.md`；本檔記「怎麼做、為什麼這樣、踩過哪些坑」。
> 完成時間：2026-06-09，issues 001–007 全數實作完畢並上線。

## 一、整條 pipeline（怎麼從 PDF 到網站）

```
pdfs_期末/*.pdf
  │  uv run python scripts/extract.py --input <檔1> <檔2> ... --output data/raw_questions_期末.json
  ▼  （extract.py：純函式 parse_questions_from_text；雙欄偵測 is_true_two_column；NON_TARGET 截斷）
data/raw_questions_期末.json
  │  uv run python scripts/classify.py --semester final
  ▼  （claude -p sonnet 批次分類 unit3/4 + 8 subtopic；無官方答案者 AI 補 + needs_review；去重/驗證走 data_helpers）
data/classified_questions_期末.json
  │  uv run python scripts/build_review_list.py   → review_期末_ai答案.md（AI 答案複查清單）
  │  uv run python scripts/build_questions.py      → 合併進 docs/questions.json（含 id 去碰撞、空選項過濾）
  ▼
docs/questions.json（期中602 + 期末；網站讀取）
  ▼  docs/index.html（期中/期末切換 + subtopic 練習，預設停期末）
```

期中同一套：`extract.py`（無參數＝讀 docs/，但期中 PDF 在 `pdfs/`，用 `--input pdfs`）＋`classify.py`（無 `--semester`＝mid）。

## 二、最容易踩的坑（務必先讀）

1. **雙欄污染**：答案卷/題目卷常是雙欄。`extract.py` 對每頁判斷是否雙欄（`is_true_two_column`）。
   - 安和答案卷第 2 頁右欄是「配合題答案欄」（`（D）(1) 25%` 型），曾被判為非雙欄→整頁逐行交錯，污染選擇題選項。已加偵測（matching_ans pattern）修掉。
   - 殘留：頁首/頁尾（「第 N 頁」「第 2 次定期評量…」）偶爾混進題目尾巴或選項。桃子腳112 mu11 的題幹甚至被截成「確？」、選項④混入頁尾——**這類「題幹/選項被雙欄打散」是目前最大的資料品質弱點**，靠 review 抓。
2. **NON_TARGET 截斷**：`NON_TARGET_SECTION` regex 界定「選擇/是非大題」的結尾。期末新增了 `填一填`、`根據題[目意]`。改這條 regex **會影響期中**（桃子腳112 期中也有「根據題意」），改完務必跑 `uv run pytest`（`tests/test_regression.py` 比對期中 golden `data/raw_questions.json`）。
3. **答案卷 ≠ 都能直接抽官方答案**：
   - 安和×4：格式A（括號內已填答案）→ 直接抽到官方答案。
   - 桃子腳112 答案卷：答案以**紅字標在選項文字上**、括號是空的 → 抽不到。改用「題目卷 + AI 補答案」。
   - 桃子腳110 答案卷：**純掃描圖片**（chars=0）→ 抽不到，改用題目卷 + AI 補答案。
   - 偵測圖片 PDF：`pdfplumber` 該頁 `extract_text()` 回空字串、`page.images` 有東西。
4. **claude -p 批次逾時**：sonnet 一批 30 題偶爾逾時（3 retry 全失敗→整批標 none，會「假性 none」吃掉題目）。已把期末 batch 降到 15、timeout 300s。重跑後若看到 `classify_reason=="分類失敗"` 的 none，就是逾時造成，要重跑。
5. **id 去碰撞**：考卷內偶有重複題號（安和110 MC 有兩個 18）。`build_questions.py` 的 `unique_id` 會對撞號加 `-2`，否則網站 `questionMap`（id→題）會默默吃掉一題。
6. **空選項 MC**：圖片型選擇題（選項是圖示、文字空白）會有「4 個空字串選項」。過濾條件是**有效（非空）選項 < 2**，不是 `len(options) < 2`。
7. **比對 reviewer 結果時 k↔題目要對齊**：review 用 `k = f"{source_stem}#{section[:2]}{number}"`。我曾把截斷顯示的 q 文字跟錯 k 配對，導致分歧報告標錯題號——**比對一律從 classified 的完整資料用 k 去查，不要用截斷過的暫存欄位**。

## 三、可切換學期設定（模組C）

`scripts/classify.py` 的 `SEMESTERS` dict：`mid` / `final` 各一份 config（system_prompt、valid_units、valid_subtopics、model、batch_size、IO 路徑）。核心分類邏輯與學期無關。要加新學期＝加一筆 config。**期中 prompt 一字不能動**（regression）。

純函式測試：`tests/test_parser.py`（parser）、`tests/test_data_helpers.py`（去重/答案合併/驗證）、`tests/test_classify_config.py`（config）、`tests/test_regression.py`（期中萃取 golden，PDF 不在時自動 skip）。`uv run pytest`。

## 四、AI 答案的可信度與複查

- 哪些是官方 vs AI：classified 的 `needs_review`（true=AI 補）；或看 `source`（安和_答案=官方，民權/桃子腳_題目=AI）。`docs/questions.json` **不帶** needs_review，只能靠 source 區分。
- `review_期末_ai答案.md`：所有進題庫的 AI 答案題清單，供人工核對。
- **單票盲審（review A）**：用 workflow 開 N 個 agent，各讀 `data/_review_blind.json`（去掉答案的盲版）盲答，與記錄比對只回分歧。123 題只 4 分歧（96.7% 一致），其中 2 題其實 AI 對、1 題 AI 可能錯（桃子腳110 mu16 漁業氣象，海水溫度才對）、1 題題幹壞掉。結論：sonnet 補答案整體可信，弱點在「題目被雙欄打散」而非答案判斷。
  - workflow 傳大量資料給 agent 的解法：**讓 workflow agent 自己用 Read 讀檔**（已驗證可讀專案檔），args 只傳小東西（如 chunk 範圍）；不要把幾十 KB 塞進 args（巢狀陣列還會傳成非陣列導致 `.map` 爆掉）。

## 五、網站（docs/index.html，單檔零依賴）

- localStorage key `aiden_study_v2`：`{challenge, errorBank, stats, semester}`。
- `challenge` 用**複合鍵**：整單元 `"3"`、subtopic `"3/動物分類"`（`challengeKey(unit, subtopic)`）。
- 期中/期末切換：`SEMESTERS`（mid=units1,2 / final=units3,4，含 subtopics）；`state.semester` 記住選擇，**預設 final**。
- 錯題模式 `quiz.unit` 是**單元陣列**（`[3]` 或 `[3,4]`），`getErrorBank` 支援陣列；錯題維持單元層級（不分 subtopic）。
- PWA：`apple-mobile-web-app-capable` + `manifest.json`（display standalone）→ iPad 可加到主畫面當 app；無 icon（iOS 用頁面快照）。
- 改完務必本機 `uv run python -m http.server` + 瀏覽器（或 Playwright）手動測；JS 無自動化測試。
