# 題幹列舉條列換行 — 設計稿（2026-06-15 grill-me 對齊，定案做法 B）

對應 CLAUDE.md 快速參考 待辦 ⑤。**狀態：已完成上線（2026-06-15，merge commit 7a8d900＋push）。** 純函式 `applyEnumBreaks` 在 `docs/index.html` 的 `<enum-break-pure>` 區塊，測試 `tests/test_breaks_pure.mjs`（12/12）。全庫審計剛好 10 題斷行、全為真列舉、0 假陽性；Playwright live 實測 MC＋fill 分行正確。

## 為什麼（rationale）

題幹遇到「並列列舉」內容時全擠一段，小三難掃讀。例：

- 桃子腳112 社會 mc_8：「花爸花媽…他們（甲）搭乘大眾運輸…（乙）但是他們都吃不完…（丙）花媽自備水壺…（丁）柚子忘記帶水壺…（戊）自備不鏽鋼餐盒…」哪些屬綠色消費？
- 自然 mc（鴿/馬/狗）：「甲.頭、乙.翅膀、丙.尾羽、丁.軀幹、戊.尾巴…」
- 自然 mc_14：「(甲)開花、(乙)結果、(丙)長高…正確順序是？」
- 填充 math：「(甲)蚵仔煎：…(乙)烤香腸：…」

現況：填充題已有自己的換行邏輯（；、半形 (N) 子題、`甲：`、答槽標籤、空格 chip；index.html renderQuiz isFill 分支），但漏掉「括號式 (甲)(乙)」；選擇/是非題完全沒有任何換行。

## 定案做法 B：純前端 regex 換行（零新檔、零 pipeline）

比照填充題現有的 runtime 換行做法，在 render 時對題幹偵測列舉標記、在標記前插 `<br>`。**純顯示、不動題庫資料、不存任何新檔**。收新題自動生效，無需記得重跑任何流程。

### 為什麼是 B 而不是重型 pipeline（避坑記憶）

本功能曾對齊一套重型設計（鬆 regex 撈候選 → 2 輪 agent 對抗審 → 烤進 `docs/breaks.json` → 獨立 pass）。**兩個獨立 reviewer agent 一致否決**，理由成立、已採納：

1. **agent 必要性的前提不成立**。當初推「非 agent 不能分語意」，怕的是「甲.頭（列舉）vs 大甲(地名)/甲班/甲袋/胖丁(人名)」。但這些假陽性**都是單一序數字＋後接漢字**——偵測規則本來就要求「**≥2 個標記**」且「**裸序數需後接標點**」，它們根本進不了候選。真正會誤撈的只剩「用**甲、乙、丙**三個容器」這種句中代號指稱，全庫個位數。
2. **規模不配**。真正要分行的乾淨列舉只有約 **11–12 題**。為十幾題養「2 支腳本＋batch 目錄＋獨立 json＋2 輪 agent＋pytest」違反「單人維護、優先簡單」。
3. **死流程風險**。專案無 CI/Makefile，獨立 pass 一年觸發一次、又不影響作答，收新題不會有人記得重跑 → 靜默失效。B 在 runtime 自動生效，免疫此問題。
4. **後果低、可逆**。純顯示，誤斷不影響作答，家長一眼看到即可回報。用這個特性換掉整條重型驗證，對單人維護划算。

（被否決的重型設計細節保留在 git 歷史，勿再提案。）

## 核心機制（B 的偵測規則）

抽成純函式 `applyEnumBreaks(stem)`，吃**已切選項後**的題幹（renderer 已算好的 displayText），回傳插好 `<br>` 的字串。

**標記家族（只認 CJK 序數，不收數字/拉丁/羅馬）**：
- 括號式：`（甲）`／`(甲)`（全或半形括號包單一 CJK 序數 甲乙丙丁戊己庚辛壬癸）。
- 裸式＋定界符：`甲.`／`甲．`／`甲：`／`甲:`／`甲、`（序數緊接標點）。

**斷行規則（防誤斷）**：
- 題幹須含 **≥2 個標記**才啟動。
- 在標記前插 `<br>`，但**只斷「後面有實質內容」的標記**：標記後緊接另一個標記（無內容）的不斷。→ 這條把「用甲、乙、丙三個容器」（甲、乙、丙互相緊鄰、無內容）整題排除，不誤斷。
- 收尾把連續 `<br>` 併一個、去開頭 `<br>`。

**為什麼不怕誤斷選項答案串**：MC 在 render 時早已切掉選項（index.html:1331-1334），`①甲乙丁②乙丙戊` 那種答案串在 displayText 裡根本不存在。TF 無選項、fill 用空格作答，皆無此風險。

**字串安全**：插 `<br>` 用 `split(marker).join("<br>"+marker)`（純字串，不是 RegExp），`(甲)`/`甲.` 含正則特殊字元也不會炸。

## 落地（涵蓋三題型）

- **MC/TF**：在切選項後（index.html:1334 之後）對 displayText 套 `applyEnumBreaks`。
- **fill**：在既有 fill 換行**之後**、chip 化**之前**（約 index.html:1354 與 1356 之間）套，補上填充漏掉的括號式 (甲)；與既有 `甲：`/`；` 換行的相鄰 `<br>` 由收尾收合。chip 只認全形 `（Ｎ）` 數字、與列舉 `（甲）` 字符集不重疊，零干擾。
- 純函式框成 `// <enum-break-pure> … // </enum-break-pure>` 區塊（sentinel 不可與 `backup-pure` 撞），不引用 DOM/全域。
- 測試 `tests/test_breaks_pure.mjs`：比照 test_backup_pure.mjs 用 `new Function` 抽區塊。**測試案例必含**：
  - 真列舉（桃子腳112 mc_8 五標記、自然 甲.頭乙.翅膀、民權110 (甲)開花、海佃110 （甲）漢人）→ 正確斷。
  - 假陽性反例（用甲、乙、丙三個容器；單標記題；甲班/甲袋）→ 不斷。
  - 連續 `<br>` 收合、開頭 `<br>` 清除。

## 實作骨架

```js
// docs/index.html
// <enum-break-pure>
function applyEnumBreaks(stem){
  const MARK = /(?:[（(][甲乙丙丁戊己庚辛壬癸][）)]|[甲乙丙丁戊己庚辛壬癸][.．：:、])/g;
  const hits = [...stem.matchAll(MARK)];
  if(hits.length < 2) return stem;
  // 只斷「後面有實質內容」的標記（後面緊接另一標記＝無內容，不斷）
  // 實作：逐標記判斷下一個字元是否又是標記起點；細節以 node 測試釘死
  // …回傳插好 <br>、收尾 .replace(/(<br>){2,}/g,"<br>").replace(/^<br>/,"")
}
// </enum-break-pure>

// renderQuiz：MC/TF 在切選項後套；fill 在既有換行後、chip 前套
```

## 尚待決定 / 已知不處理

- **萃取破損題**（如民權112 mc_8 `甲 . 乙. . 丙 . 丁 .`）：標記殘缺、不處理，留給萃取修正（比照百葉箱）。
- **大墩百葉箱 (1)(2)(3)(4)**：選項滲漏非列舉，不在範圍。
- **數字 (1)(2)**：填充子題編號/選項滲漏，不納偵測（填充已自己處理）。
- **拉丁/羅馬 (a)(i)**：現庫 0 筆，不做（YAGNI），真出現再加。
